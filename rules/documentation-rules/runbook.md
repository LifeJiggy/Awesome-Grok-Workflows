# Runbook Documentation Rules

## Structure
1. **Purpose** - What this runbook covers
2. **Prerequisites** - What's needed before starting
3. **Steps** - Numbered, actionable steps
4. **Verification** - How to confirm success
5. **Rollback** - How to undo if something goes wrong

## Common Runbooks
1. **Incident Response** - How to handle incidents
2. **Deployment** - How to deploy new versions
3. **Rollback** - How to revert deployments
4. **Scaling** - How to scale services
5. **On-Call** - What to do when on-call

## Format
1. **Clear Title** - Descriptive and specific
2. **Audience** - Who this is for
3. **Frequency** - How often this is run
4. **Time Estimate** - How long it takes
5. **Owner** - Who maintains this

## Maintenance
1. **Test Regularly** - Keep runbooks current
2. **Update After Incidents** - Learn from issues
3. **Version Control** - Track changes
4. **Review Quarterly** - Ensure accuracy
5. **Feedback Loop** - Let authors know issues

## Tools
1. **Markdown** - Portable, readable format
2. **Wiki** - Centralized, searchable
3. **Markdown Here** - Easy formatting
4. **Diagrams** - Visual aids when helpful
5. **Code Blocks** - For commands and configs
