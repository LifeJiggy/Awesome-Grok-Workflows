# API Documentation Rules

## Documentation Structure
1. **Overview** - What the API does
2. **Authentication** - How to authenticate
3. **Endpoints** - All available endpoints
4. **Examples** - Request/response samples
5. **Errors** - Error codes and messages

## Endpoint Documentation
1. **HTTP Method** - GET, POST, PUT, DELETE
2. **URL Path** - Full endpoint path
3. **Parameters** - Path, query, body params
4. **Responses** - All possible responses
5. **Examples** - Sample requests and responses

## Best Practices
1. **Code Samples** - Multiple languages
2. **Interactive Docs** - Try-it-out functionality
3. **Version History** - Track changes
4. **Searchable** - Easy to find endpoints
5. **Maintained** - Updated with code changes

## OpenAPI/Swagger
1. **Complete Spec** - All endpoints documented
2. **Accurate** - Matches actual behavior
3. **Validated** - Passes OpenAPI validation
4. **Descriptive** - Clear descriptions
5. **Examples** - Sample values for all fields

## Anti-Patterns
1. **Outdated Docs** - Don't match current API
2. **Missing Info** - Key details omitted
3. **Inconsistent** - Different formats
4. **No Examples** - Hard to understand
5. **Poor Organization** - Hard to navigate
