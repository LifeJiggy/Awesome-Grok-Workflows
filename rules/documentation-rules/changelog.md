# Changelog Documentation Rules

## Format
1. **Reverse Chronological** - Newest first
2. **Categorized** - Features, Bug Fixes, etc.
3. **Dated** - Release date for each entry
4. **Versioned** - Clear version numbers
5. **Linked** - References to issues/PRs

## Content
1. **New Features** - User-facing additions
2. **Bug Fixes** - What was fixed
3. **Breaking Changes** - Migration steps
4. **Deprecations** - What's being phased out
5. **Under the Hood** - Internal changes

## Style
1. **Clear and Concise** - Easy to scan
2. **Consistent Format** - Same structure throughout
3. **User-Focused** - What matters to users
4. **Actionable** - Users know what to do
5. **Translated** - In user language if needed

## Automation
1. **Keep a Changelog** - Based on Keep a Changelog
2. **GitHub Releases** - Auto-generate from tags
3. **Conventional Commits** - Auto-generate from commits
4. **Version Tagging** - Clear release tags
5. **Semantic Versioning** - Clear versioning scheme

## Best Practices
1. **Write for Users** - Not developers
2. **Include Examples** - Show how to use new features
3. **Migration Guide** - For breaking changes
4. **Deprecation Timeline** - Clear timeline
5. **Thank Contributors** - Credit community
