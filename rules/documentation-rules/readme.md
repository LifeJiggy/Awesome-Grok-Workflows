# README Documentation Rules

## Required Sections
1. **Project Name** - Clear title with tagline
2. **Description** - What and why in 2-3 sentences
3. **Quick Start** - 3-5 steps to first run
4. **Features** - Key capabilities
5. **Installation** - Detailed setup steps
6. **Usage** - Common use cases with examples
7. **Contributing** - How to contribute
8. **License** - Open source license

## Recommended Sections
1. **Screenshots** - Visual demonstration
2. **Architecture** - High-level design
3. **API Documentation** - For APIs
4. **Testing** - How to run tests
5. **Deployment** - How to deploy
6. **Roadmap** - Future plans
7. **FAQ** - Common questions
8. **Support** - How to get help

## Best Practices
1. **Keep Updated** - Reflect current state
2. **Use Markdown** - Portable format
3. **Include Badges** - CI status, coverage, etc.
4. **Link to Docs** - Don't repeat
5. **Code Examples** - Working, up-to-date

## Structure
1. **Executive Summary** - Top of file
2. **Getting Started** - First steps
3. **Documentation** - Links to more
4. **Advanced** - For power users
5. **Community** - Contributing, support

## Anti-Patterns
1. **Too Long** - Should fit on one screen
2. **Outdated** - Doesn't match code
3. **Missing Info** - Can't get started
4. **Too Technical** - Hard to understand
5. **No Examples** - Can't see how to use
